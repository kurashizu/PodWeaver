这里的“快照”（Snapshot），是理解 Git 最核心的概念。 如果说 SVN 是一个“变更日志”，记录着“从A到B发生了什么变化”，那么 Git 就是一个“时间胶囊”。 它存储的不是“如何从上一个版本变化到当前版本”，而是“在某个特定时间点，整个仓库的完整样子是长什么样子的”。 你看，这个区别有多大？ 这就像你在做历史研究。 SVN 给了你一份详尽的“修改说明书”，告诉你哪个文件在什么时候被改了多少。 而 Git 给了你一个“定格的现场照片”，它直接告诉你，在那个时间点，这个文件、那个文件的内容，是完全固定的、完整的样子。 这种快照机制，带来了极其强大的能力。 因为它存储的是完整状态，所以 Git 在进行任何历史回溯、任何分支切换的时候，它的计算效率和逻辑的清晰度，都达到了一个非常惊人的高度。 而这一切，都建立在“分布式”这个架构之上。 当一个项目是分布式的时候，它就不存在一个唯一的“权威中心”了。 每一个参与协作的开发者，他的本地仓库，实际上都是这个“项目历史”的一个完整、可操作的副本。 你想想这个场景，如果那个中央服务器——那个叫“远程仓库”的服务器——突然因为什么原因，比如硬件老化，突然崩溃了。 在 SVN 的世界里，大家可能就卡住了。 但在 Git 的世界里，你的团队成员依然可以继续工作。 为什么？ 因为你的本地仓库里，早就包含了完整的项目历史和所有分支的快照。 你们可以互相协作，互相同步，你们的“真理”不再依赖于某一台机器。 这带来的，是极高的“高可用性”。 这对于追求系统健壮性的咱们来说，简直是天作之合。 但这还不是全部。 分布式模型带来的另一个巨大的优势，是它彻底解放了我们的“开发灵活性”。 在中心化系统中，如果你想做一个激进的实验性修改，比如，我想看看如果把这个模块完全重写一遍会是什么样子？ 在 SVN 里，你通常需要创建出一个新的“分支”，这个分支的创建和维护，往往需要在中央服务器上进行，这会涉及到服务器资源的开销，流程感会比较重。 但在 Git 的分布式世界里，创建分支，就像是你在自己的本地桌面上，复制一份文档，然后开始在这个副本上自由创作。 这个“本地分支”是完全在你自己的机器上完成的，它几乎是瞬时的、零成本的。 你可以在本地，在一个完全隔离的、安全的沙箱里，疯狂地尝试、修改、破坏，直到你觉得这个想法是完美的。
