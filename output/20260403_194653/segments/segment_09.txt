然后呢，还有一个开发分支，通常叫 Develop，这是所有新功能和修复的“试验田”。 所有的开发者都在这里工作。 当一个新功能准备好了，它会从 Develop 分支分出一个“特性分支”，Feature Branch。 这个特性分支就像是为一个特定任务量身定制的“沙盒环境”。 你在里面构建功能，测试它，确保它在自己的小世界里是完美无瑕的。 等特性分支完成了，它会重新合并回 Develop。 这就像是，经过严格测试的部件，被送回总装车间，准备进行下一次集成。 而当整个版本准备发布的时候，我们还会从 Develop 分支分出一个“发布分支”，Release Branch。 这个分支的使命非常明确，那就是“打磨”。 在这个阶段，我们不会再添加新功能了，我们只做最后的技术校验、版本号的确定、Bug 的最后修复。 然后，一旦发布完成，这个 Release 分支会同时合并回 Master（标记为发布），也合并回 Develop（确保修复被保留）。 最后，如果生产环境出了紧急的 Bug，我们不会去重新激活整个开发流程。 我们会直接从 Master 分支分出一个“热修复分支”，Hotfix Branch。 这个分支的目标只有一个：最快速度修复生产环境的漏洞，然后迅速合并回 Master，同时也合并回 Develop，保证这个修复不会丢失。 Git Flow 的优点是它的结构性极强，它提供了一个非常清晰的、可预测的流程图。 它就像是给你的整个软件生命周期，画出了一张极其详细的、分阶段的城市规划蓝图。 但是，咱们也得客观地看它的“代价”。 它的流程是相对“重”的。 每一次功能迭代都需要经历一个完整的、多阶段的分支创建、合并、发布、热修的流程。 对于那些迭代速度极快、追求极致敏捷的团队来说，Git Flow 的这些流程节点，可能会变成一种拖沓。 所以，这就要引出第二个哲学——Trunk-Based Development，简称 TBD，主干开发。 如果说 Git Flow 是一个规划详尽的、分阶段的“城市建设”，那么 TBD 就是一个持续不断的、精益求精的“施工现场”。 在 TBD 中，核心思想是：尽可能少地创建长期分支。 所有人都应该直接、或者只在极短生命周期的分支上，向主干（Trunk，也就是我们常说的 Master 或 Main）提交代码。 这个模式极其依赖于团队的纪律性和自动化测试。
